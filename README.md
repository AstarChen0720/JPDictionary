上帝的建築工地總結（JPDictionary 專案）
上帝（開發者）： 你是最高意志，掌管一切，負責編寫「規則」與「要求」。

網頁（房子）：

HTML： 房子的硬體設施（牆壁、屋頂、窗戶）。

CSS： 房子的軟裝與材質（顏色、壁紙、甚至是特定材質的牆）。

JavaScript： 自動觸發的規則（例如有人進廚房就開燈）。

瀏覽器（工人派遣公司）： 負責在各種環境（地形）下，按照你的要求蓋出一模一樣的房子，並將複雜的硬體轉化為 DOM（建築圖） 方便管理。

React（主管）： 因為工人太笨，上帝請來主管。上帝只需對主管下簡單的命令，主管會負責處理繁瑣的細節。

Vite（工頭/翻譯官）： 主管（React）講的話工人聽不懂，需要工頭翻譯。且工頭很聰明，會實行「局部施工」，改窗戶就只改窗戶，不拆整棟房。

Node.js（基礎設施）： 工地現場的平地、自來水與電力系統，沒有它，任何工頭或工人都無法開工。

npm（物流中心/服務商）： 所有在 Node.js 基礎上運作的東西、家具、甚至是請主管（React）或工頭（Vite）的勞務服務，都從這裡叫貨。

原型屋（Vite 預設專案）： 工頭一來就先用他帶來的工具和 React 主管，在 localhost:5173 這片工地上搭好了一個樣品屋給你測試。

LV3要分幾個小步驟:先決定要跟哪家中央廚房合作,並簽約,然後約定好我們怎麼下單後他要煮什麼菜品回來給我們(餐品製作SOP合約),最後將他送回的菜裝進我們的便當後出餐
  1. 決定api(決定要跟誰合作)
  2. 制定規則(約定好合約)並測試
  3. 自動傳送和分類他回覆的指令(新的點餐SOP)
  4. 顯示出來(出餐)

LV4要分的步驟:現在每個便當都有很多字條顯示便當名或菜名,但是有些名子太刁鑽不會念,客人要問這怎麼念,餐廳內人員會念但是不標準(也不是很會XD)所以現在。要將顧客的問題轉寄去專業播音公司請他們幫我們唸(他們公司內有專業人員可以念的又標準又自然),然後播音公司會將結果放CD寄回,再將這CD放入撥放器撥給客人聽(這次不用駐點人員,因為這次需求的服務比較簡單只要寄信就可以了)
  1. 先決定要跟哪家播音公司合作.並拿到會員卡(apikey)
  2. 設定好播音SOP
     1. 將客人要問的字跟委託書(例如讀音要怎樣,這是日文還英文)一起寄給他們公司
     2. 他會寄回一cd,將其放入撥放器播放出聲音
  3. 將可以讓客人詢問的字條旁邊放一個小按鈕,按下他相當於問這怎麼念,而這時就會處發播音SOP
  4. 將拿到的CD先放好不要播完就丟掉(緩存),這樣客人要重複問時就不用我們再寄信去問

LV5要分成兩部分:資料同步,用戶認證(註冊/登入)和資料隔離
資料同步:就像是將資料複製一份到倉庫,即是店拆了(unload)下次再蓋時,只要再去倉庫拿就可以有之前的資料
  1. 找要合作的倉儲公司,並且約定好倉庫要怎樣存東西(例如跟他說我到時候會一次寄蘋果,香蕉,芭樂,然後你幫我一種放一堆,而不是全都直接丟倉庫),然後拿到會員卡,請駐點人員過來店裡
   2. 寫增刪改查SOP,因為所有要跟倉庫的互動都能簡化成"增刪改查"四種動作,但都要透過倉儲駐點人員,且有些還有包包裹,拆包裹等步驟,所以定成SOP比較方便
    2.1向倉庫增加東西的SOP
    2.2將倉庫某東西刪除的SOP
    2.3將倉庫某東西修改的SOP
    2.4向倉庫查取東西的SOP
  3. 改點餐SOP,讓每次做好便當要出餐時就將這出餐紀錄,讓倉儲駐點人員寄去倉儲公司存進倉庫存起來(執行向倉庫增加東西SOP)
  4. 向倉庫同步資料,讓開店時(或換不同客人時(以後在改)),向倉庫查取客人資料,並將目前的資料撕掉,將查到的抄上去
  5. 刪除倉庫資料,讓歷史清單的右上有一個叉,按下就會刪除這單字卡,同時刪除倉庫內的同一筆資料(執行將倉庫某東西刪除SOP)
  6. 修改倉庫資料,讓便當清單的每一單字卡的右下角有一個"修改"的按鈕,按下後就可以修改文字,改完後倉庫內的資料也會同步修改(執行將倉庫某東西修改SOP)
用戶認證(註冊/登入)和資料隔離:

請駐點人員進駐： 在 App.jsx 裡引入他們的工具包（安裝套件）。

簽署入庫契約： 設定好你的許可證（API Key），讓駐點人員知道他服務的是哪家店。

改寫點餐 SOP： 以前 takeOrder 是把內容抄在筆記本（useState），現在除了抄筆記本，還要多一個動作：「交給駐點人員寄回總部」。

開店領貨流程： 在便當店剛開門時（useEffect），主動請駐點人員問公司：「有沒有我們店之前的存貨？」有的話就通通搬回來放到筆記本裡。
